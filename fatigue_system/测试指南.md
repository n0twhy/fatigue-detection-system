> 这份文档告诉你**要测什么、怎么测、数据怎么统计、最后交什么**。
> 软件基本操作（打开/校准/记录按钮在哪）见 `运行说明.md`，本文不重复。
> 测试全程**不需要写代码**。

## 一、总览：要交付的测试内容

对应课程任务书"步骤5：系统测试与结果分析"，需要完成：

1. **6 个基础场景测试**（必做）：正常 / 闭眼 / 频繁眨眼 / 打哈欠 / 低头 / 综合疲劳。
2. **鲁棒性附加测试**（建议做，报告加分）：戴眼镜、暗光、侧脸、多人入镜。
3. 每个场景产出：**一段测试视频 + 两个 CSV**（明细+汇总）+ 结论一行
   （检出/漏检/误报、延迟多少秒）。

---

## 二、录制测试视频的要求

- 每个场景**单独录一段**，30~60 秒，mp4/avi 均可（手机/电脑摄像头都行）。
- **光线充足、脸正对镜头、占画面 1/3 以上**——光线差会同时拖慢帧率、影响眼部
  和心率检测，测出来的是环境问题不是系统问题。
- 每段开头留 **5 秒清醒正视**（方便对齐时间轴：动作从第几秒开始，记下来）。
- 动作脚本建议：

| 场景 | 动作 | 期望结果 |
|---|---|---|
| ① 正常 | 全程清醒自然看镜头，正常眨眼 | 全程"清醒"，**零报警**（考察误报率） |
| ② 闭眼 | 第 10s 起持续闭眼 15~20s | 约 2s 内直接重度+报警（微睡眠硬规则）；PERCLOS/最长闭眼上升 |
| ③ 频繁眨眼 | 刻意快速眨眼（每秒 1 次左右）30s | 眨眼率 >40 次/分，等级到轻/中度 |
| ④ 打哈欠 | 每 15s 打一次大哈欠（张嘴 ≥2s），共 3 次 | 哈欠计数 3，等级到轻/中度 |
| ⑤ 低头 | 第 10s 起持续低头看膝盖 | 头部状态=低头，等级到中度 |
| ⑥ 综合疲劳 | 闭眼+哈欠+低头轮番/叠加 | 重度 + 弹窗 + 响铃报警 |

## 三、每段视频的标准测试流程（每段约 2 分钟）

1. 打开软件 → 「打开视频文件…」选这段视频。
2. **点「开始记录」**（在视频开始播后马上点）。
3. 视频**播完会自动停**（显示"▶ 视频播放结束"）。
4. 点「**停止并保存**」→ 输出目录生成两个 CSV（exe 版在 exe 旁 `outputs\`）：
   - `fatigue_log_日期时间.csv` —— 每秒一行的明细；
   - `session_summary_日期时间.csv` —— 本段汇总（各等级时长占比、报警次数、平均帧率）。
5. **马上把两个 CSV 重命名**加上场景名（如 `场景2_闭眼_明细.csv`），避免混淆。
6. 在记录表（下节）里填一行结论。

> 摄像头实测（不用视频文件）流程相同，但**先做 30s 校准**再测，结果更贴合本人。
> 视频回放测试可不校准（校准需要本人在镜头前实时配合）。

## 四、指标怎么统计（写报告用）

打开 CSV（Excel 直接开，不会乱码），按下面口径统计：

### 4.1 每个场景的判定正确性
- 看**明细 CSV** 的 `level` 列（0清醒/1轻度/2中度/3重度）与 `alarm` 列（0/1）。
- **检出**：动作时段内，等级是否达到"期望结果"列的等级？
- **检出延迟**：从"动作开始的秒数"到"等级首次达标的行"差几秒。
  （②闭眼场景另看：从闭眼开始到 alarm=1 差几秒，应约 2s。）
- **误报**：正常时段（含场景①全程）出现 alarm=1 即记一次误报。

### 4.2 汇总表格（报告模板）

| 场景 | 期望等级 | 实际最高等级 | 检出(√/×) | 延迟(s) | 误报(次) | 备注 |
|---|---|---|---|---|---|---|
| 正常 | 清醒 | | | — | | |
| 闭眼 | 重度+报警 | | | | | |
| 频繁眨眼 | 轻/中度 | | | | | |
| 打哈欠 | 轻/中度 | | | | | |
| 低头 | 中度 | | | | | |
| 综合疲劳 | 重度+报警 | | | | | |

- **准确率** = 检出场景数 / 6；**误报率**看场景①与各场景正常时段。
- **实时性**：汇总 CSV 里的"平均帧率"应 ≥15fps（视频回放一般 25~30fps）。
- 特征级误差（可选加分）：③场景数你实际眨眼次数 vs `blink_rate`；
  ④场景对照 `yawn_count` 是否=3。

### 4.3 鲁棒性附加测试（建议）
同一动作（如闭眼）分别在 **戴眼镜 / 暗光 / 侧脸 45° / 背景有第二个人** 下各录一段，
记录哪些条件下检出变差——如实写进报告"应用局限"，这是评分点（分析深入）不是扣分点。

## 五、常见问题判断（先自查再报 bug）

| 现象 | 先检查 | 说明 |
|---|---|---|
| 闭眼检不出 | 是否做过校准？光线？ | 摄像头实测**务必先校准**；视频回放用默认阈值，弱信号视频可能欠佳 |
| 心率显示"—" | 光线、脸是否稳定、是否满 10s | 算不准时宁可不显示，属设计行为 |
| 生理子分显示 0.00 | 不是 bug | 它是"心率下降幅度"，不困时就是 0 |
| 眨眼率开头偏高后回落 | 不是 bug | 统计窗口未填满的正常现象，30~60s 后趋稳 |
| 一直不报警 | 动作保持得够久吗 | 报警需"持续重度约 3s"（防误报），综合场景动作要叠加保持 |
| 帧率很低 | 光线 | 暗光会让摄像头供帧掉到 ~8-15fps，开灯即恢复 |

真 bug 请附：场景视频 + 两个 CSV + 现象描述（第几秒、期望 vs 实际），发给开发同学。
