### M4 实时 rPPG 辅线 ✅
- `core/rppg_realtime.py`（§6.6）：`RealtimeRPPG(fps, cfg)` 滚动缓冲(buffer_sec) → 按 ts **线性重采样到均匀网格**(相机供帧不均匀，否则 FFT 频率轴失真) → **POS**(Wang 2017 核心移植自 `unsupervised_methods/methods/POS_WANG.py`，输入改为逐帧 ROI 均值迹) → 带通 0.7-3Hz → **FFT 零填充**取峰 = HR。`estimate()` 内部按 `rppg.update_interval_sec` 缓存，帧循环高频调用无压力。
- 接线（`main_window.py`）：每帧 `rppg.update(ff.roi_rgb, ts)` → `wf.hr/hrv` 汇入窗口特征 → `physio_subscore`(相对静息基线的**心率下降比例**)汇入融合 → 静息 HR 也喂进基线校准(`calibrator.push(ff, hr=)`)。`rppg.enable=false` 或无基线 HR 时 physio=None，融合按剩余权重归一化=退化基础模型。换源/ts回退时 rPPG 缓冲与滑窗均作废(防跨轮污染)。
- **精度定位（报告如实写，勿夸大）**：**HR 是主生理输出**——实测对 UBFC 真值(`data/Test/subject1`)逐秒 **MAE≈7.8bpm**（POS 无监督合理范围），合成 4 组已知 HR(54~100bpm@8~30fps)误差≤5bpm。**HRV(RMSSD) 仅粗略趋势**——10s 短窗+CPU 摄像头下峰定时噪声底 ~30-130ms 淹没真实 HRV，已加"频谱HR一致性+生理上限"四重门控，真实视频 45 窗中 32 窗判"不可靠"(None)、仅 13 窗出值，**不参与融合、不保证数值精度**，只作辅助显示。
- 自测全过：合成 HR 4 组 + HRV 门控 + 缓冲未满/长间断/ts回退边界(8 项)、真实视频 vs UBFC 真值(4 项)、physio 子分路径(HR降15%→满分/升高→0/在场不重归一)、M3 全回归 + GUI e2e(含rPPG) + selftest 干净退出。

## 4. 关键技术决策与踩坑（新会话必读，否则会重复踩）

1. **Python 3.8 联合类型**：规格书写的 `tuple | None`（PEP604）在 3.8 不可用，代码里统一用 `typing.Optional[...]`。
2. **cv2 ↔ PyQt5 Qt 插件冲突**：`import cv2` 会把 `QT_QPA_PLATFORM_PLUGIN_PATH` 劫持到 `cv2/qt/plugins`（其 libqxcb.so 与 PyQt5 不兼容），导致 xcb 加载失败崩溃。修复在 `app.py::_use_pyqt5_qt_plugins()`——在 import cv2 之后、建 QApplication 之前把该变量指回 `PyQt5/Qt5/plugins`。**任何新的 Qt 入口都要保证这个先执行。**
3. **中文字体**：系统无中文字体，界面会全是方块。`app.py::_setup_cjk_font()` 从 `/mnt/c/Windows/Fonts/msyh.ttc`（微软雅黑）等自动加载并设为默认字体。**注意 cv2.putText 不支持中文**，所以视频上的 HUD 用 ASCII；中文只放在 Qt 控件里。窗口标题栏乱码是 WSLg 窗口管理器(Weston)画的、与 app 字体两套，属外观问题，装 `fonts-noto-cjk` 可根治。
4. **mediapipe 受控安装**：见 §1。核心是 `--no-deps` + 手动补纯 Python 依赖，避免破坏 toolbox 的 numpy/protobuf/opencv。
5. **摄像头（WSL2 关键）**：WSL2 默认无摄像头。本机内核 `6.6.87.2` 含 `uvcvideo`/`v4l2`，可用 **usbipd-win** 挂载：Windows 管理员 PowerShell `usbipd bind --busid <id>` + `usbipd attach --wsl --busid <id>`；WSL 里 `sudo modprobe uvcvideo`。**两个必须**：(a) `/dev/video*` 属 root:video 且 660，用户不在 video 组 → 需 `sudo chmod 666 /dev/video*`（或加入 video 组）才能打开；(b) **必须 MJPG 格式**（已在 `video_source.open_camera` 里 `cap.set(FOURCC, MJPG)`）。
6. **头姿 ±180° 归一化**：solvePnP+RQDecomp3x3 常把 pitch 解成 ±180°翻转（正视时≈-167°）。`head_pose._normalize_angle` 折叠到 (-90,90]，正视≈+14°（该偏移由基线校准归零）。
7. **摄像头枚举平台感知**：`video_source._candidate_camera_indices` 在 Linux 只认真实 `/dev/video*` 节点（无则返回空、不盲探），避免 WSL2 无摄像头时刷一屏 V4L2 警告。
8. **refine_landmarks 试了又回退**：开启虹膜精修会把 EAR 尺度整体压低（睁眼 0.30→0.17）、打乱既有阈值与校准逻辑，实测算力代价虽近乎为零但**尺度不一致弊大于利，故保持 false**。
9. **数据集软链接**：`data/UBFC/`、`data/UBFC_20/`、`data/UBFC-rPPG_BLURRED_*/` 全是**指向 `/mnt/f/`（Windows F: 盘）的软链接，F: 当前未挂载**，故这些 vid.avi 打不开（不是编码问题）。**唯一能用的真实视频是 `data/Test/subject1/vid.avi`**（21MB，MJPG，1547帧@29.3fps）。挂 F: 命令：`sudo mount -t drvfs F: /mnt/f`。
10. **绝不对摄像头 set CAP_PROP_FPS**（§5.1 低帧率问题的真凶）：对 usbip 摄像头请求 20fps 会协商出**实际仅 ~5-6fps** 的降级模式；保持驱动默认 30fps 模式反而能拿到当前光照下的最高供帧率（实测矩阵：不设=12-13.5fps、设20=5-5.8fps、设30=12.7fps，暗光环境）。`target_fps` 只作 GUI 帧循环节拍，与相机模式无关。另注意：**暗光下自动曝光本身会压低供帧率**（白天可到 ~27fps，深夜暗光只有 ~12-15fps），这是物理限制非软件问题。
11. **摄像头采集线程必须在进程退出前停掉**：daemon 线程阻塞在 `cap.read()` 时解释器退出会触发 `FATAL: exception not rethrown` + core dump。而 `app.quit()`（如 `--selftest`）**不走 closeEvent**，故 `MainWindow` 的清理做成幂等 `_cleanup()` 并同时挂到 `closeEvent` 与 `QApplication.aboutToQuit`。新增任何退出路径都要保证走到它。
12. **时长/去抖判定一律用时间戳，绝不用"帧数"**：实际帧率会低到 ~8fps（usbip+暗光），任何隐含 30fps 假设的帧数逻辑都会失效。已踩实的两个案例（均已修，`feature_window.py`）：① 眨眼"连续≥2帧闭眼"去抖——8fps 帧间隔 125ms，正常眨眼全闭合仅 100~150ms 永远凑不满 2 帧 → **一次眨眼都检不出**；② 哈欠段时长按"首帧→末帧"少算一个帧间隔——8fps 下 1.5s 哈欠只算出 1.375s 恰好漏检。现统一"含帧间隔"语义：段 = 首个满足帧 → 其后首个不满足帧。**M3+ 写融合/报警 FSM 时同理**（`alarm_consecutive_windows` 按窗口数没问题，窗口本身是按秒定义的）。
